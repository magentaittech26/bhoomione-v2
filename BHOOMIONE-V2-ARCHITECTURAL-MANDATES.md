# BHOOMIONE V2: MANDATORY ARCHITECTURAL MANDATES & DESIGN LAWS
**Document ID**: BHOOMIONE-V2-ARCHITECTURAL-MANDATES  
**Status**: Signed, Locked & Immutable  
**Version**: 2.0  
**Authority**: Lead Enterprise Architect & DevSecOps Lead  
**Date**: June 19, 2026  

---

This document represents the absolute architectural constraints and design laws for the **BhoomiOne V2** platform. Every module, API interface, database schema, and frontend component built under this system must adhere to these ten mandates. There are no exemptions.

---

## Mandate 1: No Mock Data Anywhere
BhoomiOne is an industrial, high-integrity platform managing high-value legal land assets and transactions. 
*   **The Rule**: Under no circumstances will any user interface, API response, or background subsystem return mock databases, mock charts, simulated transactions, or static "demo" placeholders.
*   **Implementation**: Every workflow—including banking reconciliations, payment portals, document verifications, RERA audits, and agent assignments—must integrate with real database state tables or legitimate third-party API gateways.

## Mandate 2: Database-First Development
All platform feature developments must begin at the data modeling layer.
*   **The Rule**: No business modules or frontend views are to be created unless their corresponding tables, indexes, spatial constraints, and foreign key rules are declared, audited, and migrated inside PostgreSQL.
*   **Implementation**: Application layers simply reflect the core transactional database state. Forms must validate against the exact PostgreSQL restrictions (column limits, data types).

## Mandate 3: Platform Admin Unified Onboarding
The Central Platform Admin Portal (`admin.bhoomione.in`) is the singular administrative controller of the entire ecosystem.
*   **The Rule**: The super-admin console must provide comprehensive, database-backed mechanisms to onboard, manage, verify, suspend, and blacklist:
    1.  **Developers (Tenants)**: B2B entity profiles, company registrations, and subscription billing.
    2.  **Agents (Brokers/Partners)**: Independent field sales agents, agency hierarchies, and RERA broker allocations.
    3.  **Customers (Buyers/Investors)**: Public buyer directories, payment logs, and deed portfolios.

## Mandate 4: KYC Mandatory for All Stakeholders
Trust is our highest product differentiator. A fully audited identity trail prevents fraudulent transactions and double-bookings.
*   **The Rule**: No stakeholder can conduct transactions, publish projects, or access inventory without verified Identity Documents (KYC).
*   **Mandatory Upload matrices**:
    *   **Developers**: PAN, GST, certificate of incorporation, corporate bank cancelled cheque, and RERA registrations.
    *   **Agents**: PAN, Aadhaar (masked), RERA broker license, and active business bank accounts.
    *   **Customers**: PAN, Aadhaar, address verification, and verified nominee details.

## Mandate 5: Marketplace Uses a Centralized Secondary Index
The Public Discovery Marketplace (`bhoomione.in`) runs independently of deep, transactional tenant databases to achieve sub-second page performance under massive concurrency.
*   **The Rule**: All public listing queries, search inputs, filtering parameters, and relative location searches must source exclusively from a pre-compiled, optimized Secondary Index database table set (or read replica cache-aside model).
*   **Implementation**: Fast lookup queries read SVG coordinate configurations from S3, completely shielding inner execution logs.

## Mandate 6: Tenant Data is Never Queried Directly by the Marketplace
The operational integrity of private developer tenant data is non-negotiable.
*   **The Rule**: The public Marketplace is strictly forbidden from executing raw, direct SQL queries against transactional core tables (e.g. `bookings`, `invoices`, or active field `reservations`).
*   **Implementation**: Any updates to plot statuses inside the Developer ERP (e.g., booking confirmation) trigger Laravel Events that asynchronously update the central Marketplace Index and emit live WebSocket updates.

## Mandate 7: DXF Import is the Core Product Feature
BhoomiOne is defined by physical land. The ability to automatically ingest and parse industry-standard civil drawing vector layouts is our primary competitive moat.
*   **The Rule**: The core DXF Parsing Engine must run in the background, utilizing asynchronous worker execution queues to prevent application server thread-locking during large imports (layouts containing 1,000+ plots).

## Mandate 8: Plot Inventory is Autogenerated from Imports
Manual, coordinate-free grid creation is deprecated.
*   **The Rule**: All cell plots, clickable polygons, centroids, dimensions, and face coordinates inside layout views must be automatically extracted and populated from DXF geometry boundaries or SVG canvas nodes. 
*   **Implementation**: Natural AutoCAD layers map directly to database objects. The coordinates partition automatically maps centroids and boundary bounds, ensuring the digital layout represents a precise architectural copy of the physical layout.

## Mandate 9: Fully Configurable Geographic Area Units
Plotted developments across India have highly fragmented, region-specific, and localized measurement metrics.
*   **The Rule**: The core layout parsing, area calculator, and invoice engines must remain fully configurable across regional land units, supporting:
    *   **SQFT** (Square Feet - Standard B2C metric)
    *   **SQM** (Square Meters - mandatory mapping metric for Goa and Maharashtra state laws)
    *   **Acre** (Agricultural and massive township expansions)
    *   **Gunta / Guntha** (Universal agricultural metric across Karnataka and regional south states)
    *   **Bigha** (Primary land division standard in northern Indian states)
    *   **Local Units** (Cents, Hectares, or Sq Yards resolved dynamically based on project geolocation indexes)

## Mandate 10: Pricing Engine Is Unit-Agnostic
Rates inside the plotting network depend heavily on localized metrics.
*   **The Rule**: No operational loop, pricing formula, or invoice compiler is allowed to assume Square Feet as the native denominator. Rates must support flexible rate structures derived on demand:
    *   *Plot Price = Area_Value × Rate_Per_Selected_Unit* (whereSelected\_Unit can map dynamically to Guntha, Sq Yard, Acre, or Local Standard).
*   **Implementation**: Pricing indexes are completely decoupled from absolute metric systems. The database pricing ledger stores the unit configuration, the unit rate, and calculates the total plot amount directly at index runtime.

## Mandate 11: Mobile-First Strategy & Field Sales Execution Rules
Real estate agents and developer sales executives work primary on-site and in the field; they require instantaneous layout updates and availability checks during site visits.
*   **The Rule**: Although a native mobile application is NOT built in Sprint 1, all APIs, backend DTOs, and UI design decisions must be explicitly mobile-optimized and mobile-ready from Day 1.
*   **Field Sales Use Case Compliance**: The system's API endpoints and UI pathways must support the sequential rollout of agent field capabilities:
    1.  Opening assigned projects.
    2.  Viewing assigned layout structures.
    3.  Checking near-instant plot availability status.
    4.  Searching specific plot numbers with sub-second lookups.
    5.  Viewing customer verification/booking indices.
    6.  Reserving or blocking permitted plots based on workflow hierarchies.
    7.  Supporting resilient offline device caching mechanisms in later phases.

## Mandate 12: Feature-Flagged GIS Map Access and Billing Enforcements
Interactive GIS maps represent premium proprietary technology and computational load; the architectural access pathways must behave as partitioned, feature-flagged, and monetizable modules.
*   **Starter Plan GIS Limitations**: The Starter Plan is strictly forbidden from executing or loading full, interactive AutoCAD/GIS clickable layouts. By native routing control, the Starter Plan only displays:
    *   Simple plot tabular grids or block tables.
    *   Primitive plot status colored blocks.
    *   Plot number metadata, dimensions (area value), and status flags.
    *   Basic form-based booking actions.
    *   No CAD DXF/SVG uploads or complex zoom/pan spatial canvas widgets.
*   **Map Module Add-On and Higher Tiers**: The interactive, vector-based MAP VIEWER and AutoCAD CAD processing systems exist purely as a paid service add-on (standardized under billing indicators in Growth, Professional, and Enterprise packages). The system must natively isolate this through billing feature flags:
    *   `feature_maps_basic`: Render simple responsive layout viewers.
    *   `feature_maps_advanced`: Full SVG click interactions, zoom/pan tools, and layer configurations.
    *   `feature_cad_import`: Unlock async backend DXF parsers and segment validators.
    *   `feature_offline_maps`: Enable mobile/PWA SQLite spatial caches.
*   **Granular Access Control Security**:
    *   If a tenant's subscription status does not hold the active Maps feature flags, the API gateway and the frontend components must automatically block, hide, and throw security exceptions on attempts to touch CAD upload engines, advanced GIS layers, and interactive web mapping shells.
