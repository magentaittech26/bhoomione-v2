**Author**: Lead Enterprise Architect

## 1. Executive Summary

This audit assesses the core architectural readiness of **BhoomiOne V2**—India's first GIS-native plotted development operating system. Traditional real estate CRMs manage only customer touchpoints and sales status, neglecting the coordinate geometry and legal-spatial identity of land itself. BhoomiOne digitizes the land as a structured, auditable digital asset. 

To scale to **50,000+ agents, 5,000+ developers, and 1,000,000+ plots**, the engineering layer must be fully stable, highly performant, and resistant to regression. The current audit identifies high-risk areas in V1 (monolithic structures, shared static builds writing directly to backend static paths, and coupled routing) and defines the exact decoupled, micro-frontend, and robust container-driven architecture for V2.

## 2. Repository Structure Audit

The V1 repository utilized a monolithic architecture where all frontends (Marketplace, Admin, Tenant, Customer, Agent) were compiled together in a single Vite build, occasionally outputting static production bundles directly into the backend's static directory. This led to serious operational blockers on production virtual private servers (VPS), where standard Git pulls or automatic pipelines would fail due to untracked asset generation and lock-file contention.

### V2 Repository Topology (Monorepo Strategy)
The recommended layout is a highly organized, domain-segregated Monorepo structure that guarantees build isolation and shared contract enforcement without directory leakage.

```
bhoomione-v2/
├── backend-api/           # Laravel 12 API & Core Domain Logic (Port 8000 / api.bhoomione.in)
├── web-marketplace/       # Public Listing & GIS Discovery Portal (Port 5173 / bhoomione.in)
├── web-admin/             # Platform Super Admin & Billing Dashboard (Port 5174 / admin.bhoomione.in)
├── web-tenant/            # Developer ERP & Spatial Workspace Portal (Port 5175 / tenant.bhoomione.in)
├── web-agent/             # Field Sales Partner & Commission Portal (Port 5177 / agents.tenant.bhoomione.in)
├── web-customer/          # Customer Digital Property Locker (Port 5176 / portal.tenant.bhoomione.in)
├── packages/              # Reusable shared packages across frontends
│   ├── shared-ui/         # Shared component design system tokens & Tailwind base
│   ├── shared-sdk/        # Auth & Autogenerated Domain-Specific API Clients
│   └── shared-types/      # Centralized TypeScript interface definitions (Plot, Project, Tenant, etc.)
├── deploy/                # Dockerfiles, Nginx configs, Prometheus/Grafana monitoring configurations
└── docs/                  # System-wide Architecture Documentation (01--49)
```

### Component Analysis:
1. **`backend-api/`**: Segregated Laravel 12 workspace running on PHP 8.4+ and PostgreSQL 17+. Contains zero user-interface code resources. It is strictly an API Provider, processing heavy transactions, CAD parses, and coordinate translations.
2. **`web-marketplace/`**: Static SPA focused purely on SEO performance and public geospatial discovery. Contains only public viewer widgets and lead generation capture interfaces. No administrative tools are compiled here, which decreases the initial packet weight (bundle size target: `<1.2MB`).
3. **`web-admin/`**: Admin portal allowing Platform super-users to verify RERA documents, audit subscription billing, adjust plans, and process KYC. Compiled in absolute isolation from the developer database tables.
4. **`web-tenant/`**: Developer Enterprise Resource Planning (ERP). Contains extensive plot inventory tables, project phase definitions, mapping upload features, and collection aging tracking systems.
5. **`packages/shared-ui`**: Tailored UI component system enforcing design consistency across the 5 frontends. 
6. **`packages/shared-sdk`**: Centralized API gateway contracts, containing strict DTOs and autogenerated TypeScript clients mapping `/api/v1` routes.

## 3. Docker Architecture Audit

The runtime containerization framework must isolate the compute-heavy DXF parsers and the volatile database write queries from public HTTP traffic.

```
                      ┌────────────────────────────────────────┐
                      │            Cloudflare DNS & WAF        │
                      └───────────────────┬────────────────────┘
                                          │ SSL / HTTPS
                                          ▼
                      ┌────────────────────────────────────────┐
                      │        Nginx Reverse Proxy (Container) │
                      └───────────────────┬────────────────────┘
                                          │
            ┌─────────────────────────────┼────────────────────────────┐
            │                             │ Remote API Proxy           │ Static SPAs (Client Ports)
            ▼                             ▼                            ▼
┌──────────────────────┐      ┌───────────────────────┐      ┌───────────────────────┐
│     backend-api      │      │         redis         │      │    web-frontends      │
│  (Laravel/PHP-FPM)   │      │ (Queues, Cache, Sess) │      │ (Marketplace, Tenant, │
│      Port 9000       │      │       Port 6379       │      │   Admin, Agent, Cust) │
└───────────┬──────────┘      └───────────┬───────────┘      └───────────────────────┘
            │                             │
            ▼                             ▼
┌──────────────────────┐      ┌───────────────────────┐
│     postgresql       │      │     Object Storage    │
│  (Postgres 17 Engine)│      │  (AWS S3 Sec Buckets) │
│      Port 5432       │      └───────────────────────┘
└──────────────────────┘
```

### 3.1 Services Architecture
*   **`nginx` (Routing Shell)**: Operates as a single entry-point reverse proxy routing public HTTPS traffic from the domains based on HTTP host headers and path specifications.
*   **`backend-api`**: Laravel PHP-FPM container configured with separate pool directives. For scale, this is accompanied by a decoupled background task runner running **Laravel Horizon/Redis** specifically for async DXF parsing and template mailers.
*   **`postgresql` (Database Engine)**: Standard PostgreSQL v17 container. Production deployments must mount database maps to a write-through host directory (`/var/lib/postgresql/data`) to prevent transient container deletion from wiping transaction history.
*   **`redis` (Cache & Broker)**: In-memory store acting as an asynchronous message broker, session database, and API rate-limiting ledger.

### 3.2 Network Layer Protection
To prevent critical vectors (such as database brute-forces or Redis key-overwrites), we declare the network namespaces below:
1.  **`bhoomione-ingress` (External/Public)**: Connects `nginx` and the five `web-frontends` containers. Only ports `80` and `443` are exposed to the outside hosting platform.
2.  **`bhoomione-api` (Intermediate/Internal)**: Connects `nginx`, `backend-api`, and `redis`. Frontends cannot access this directory directly except via HTTP proxies managed by the central proxy container.
3.  **`bhoomione-db` (Private/Isolated)**: Connects `backend-api` to `postgresql` and `redis`. The PostgreSQL container is **strictly unexposed** on the public interface, forbidding port `5432` from public binding.

## 4. Domain & Routing Architecture Audit

Consistent multi-tenancy requires strict domain mappings mapped directly at the routing layer so that tenant IDs can be resolved transparently.

| Portal Domain Path | Target Port | Container Name | Target Security Protocol |
| :--- | :--- | :--- | :--- |
| **`bhoomione.in`** | `5173` | `marketplace-app` | Public Ingress / Read Only |
| **`admin.bhoomione.in`** | `5174` | `admin-app` | Admin Core JWT / MFA Mandatory |
| **`tenant.bhoomione.in`** | `5175` | `tenant-app` | Tenant Context JWT / B2B Auth |
| **`portal.tenant.bhoomione.in`** | `5176` | `customer-app` | Customer Self Service / Encrypted |
| **`agents.tenant.bhoomione.in`** | `5177` | `agent-app` | Mobile-Optimized Field JWT |
| **`api.bhoomione.in`** | `8000` | `backend-api-app` | Strict CORS / Rate Limited Gateway |

### Dynamic Tenant Resolution Rules (Wildcard Mapping)
Under the subdomain routing architecture, when a developer registers as `shaurya-builders`, their tenant portal maps to `shaurya-builders.tenant.bhoomione.in` or their dedicated custom domain (e.g., `plots.shauryabuilders.com`). 
1. The **Nginx reverse proxy** captures the host header.
2. If the host resolves via regex to any customer portal (e.g., `*.tenant.bhoomione.in`), Nginx passes the request to the central `backend-api` adding instructions: `X-Tenant-Domain: $host`.
3. The API tenant middleware looks up the resolved hostname in the SQL `tenant_domains` index table and binds the corresponding `tenant_id` context directly into that execution thread before running any query.

## 5. Build Isolation Audit

The BhoomiOne V2 architecture introduces **unyielding compile-time boundaries** between all frontends to guarantee absolute security and maximum page performance.

### 5.1 Verification Matrix
*   **Zero Frontend Leakage**: Under NO circumstances should any build process (e.g., `web-tenant`) output its compiled static artifacts inside the `backend-api/public/` folder. Doing so corrupts Git commits, blocks VPS deploys, and poses a major security hazard where an admin asset could end up served inside public listing endpoints.
*   **Vite Build Targets**: Every SPA compiles to its independent folder inside the workspace or a distinct CI pipeline:
    *   `web-marketplace/` ──> `/dist-marketplace`
    *   `web-admin/` ──> `/dist-admin`
    *   `web-tenant/` ──> `/dist-tenant`
    *   `web-agent/` ──> `/dist-agent`
    *   `web-customer/` ──> `/dist-customer`
*   **De-coupled App.tsx**: The V1 layout pattern of nesting marketplace components, agent calculators, and developer forms inside a giant monolithic `/src/App.tsx` is completely outlawed. Under V2, each frontend has its own simple `App.tsx` housing only routes, global providers (React Query, context engines), and the top-level layout shell. 
*   **Isolated Assets Control**: Global images, AutoCAD parsing examples, and DXF resources are moved into secure AWS S3 private buckets, guarded by IAM and accessed only via pre-signed, short-lived URLs.

## 6. Target Performance & Scalability Benchmarks

To establish BhoomiOne V2 as an institutional PropTech framework, developers must feel immediate performance feedback, particularly during heavy GIS map routing.

| Target Event Metric | Benchmark SLA Threshold | Scaling Target Capacity | Core Technological Method |
| :--- | :--- | :--- | :--- |
| **First Contentful Paint (FCP)** | `< 1.2 seconds` | 2,000,000 monthly hits | Code-splitting, Brotli compression, edge caching |
| **GIS Map Interactivity** | `< 2.0 seconds` | 100,000 concurrent plots | Pre-rendered vector SVGs, coordinate virtualization |
| **Plot Selector Click-response** | `< 100ms` | 25,000 clicks / second | Centroid bounding-box cache directly in Redis |
| **DXF Layer Parsing & Extraction** | `< 30 seconds` | 500 concurrent CAD uploads | Decoupled Python-FPM `ezdxf` queue workers |
| **API Gateway Auth Verification** | `< 150ms` | 15,000 requests / second | JWT stateless claims, local public key encryption |
| **Outstanding Ledger Aging Query** | `< 500ms` | 100M transaction database rows | Composite table range indices, materialized aggregation |
